> Документ-помощник Максиму. Здесь: суть проекта, что строим, на чём, план по этапам и **готовые промпты** для Antigravity. Промпты копируй по одному, в порядке этапов. После каждого шага — проверяй, что работает, прежде чем идти дальше.

## 1. Суть в двух словах

«Паспорт жизни» — это платный онлайн-сервис: человек заполняет форму о себе (имя, дата/время/место рождения; либо 5 важных событий жизни, если время неизвестно), оплачивает через BePaid и получает персональный PDF-«паспорт», сгенерированный ИИ.

Главное действие пользователя: **заполнить форму → оплатить → получить и скачать готовый PDF-паспорт.**

Система из трёх частей:
- **Фронтенд (есть)** — лендинг с формой, экран прогресса, экран результата.
- **Бэкенд РБ («дирижёр»)** — принимает форму, проводит оплату, дёргает польский API, собирает PDF, отдаёт результат.
- **Польский API («ИИ-цех»)** — отдельный сервис: собирает промпт, обращается к ИИ, возвращает структурированный JSON.

## 2. Что умеет (функционал)

**MVP (делаем сначала — самое сердце):**
- Форма с валидацией и логикой чекбокса «не знаю точное время» (показывает блок 5 событий).
- Оплата через BePaid с подтверждением по вебхуку.
- Передача данных в польский API и приём JSON.
- Польский API: сборка промпта по шаблону + вызов ИИ + возврат JSON.
- Генерация PDF из полученного результата и его показ/скачивание.
- Экран прогресса, страницы SUCCESS / FAIL.
- Логирование всех этапов на обеих сторонах.

**Потом (улучшения):**
- Повторная обработка «зависших» заказов кнопкой/по расписанию.
- Админ-страница со списком заказов и логами.
- Письмо пользователю с PDF на почту.
- Хранение результатов в полноценной облачной базе, бэкапы.
- Перенос фронта на сборщик (Vite), если CDN-вариант станет неудобным.

## 3. Как этим пользуются (сценарий и экраны)

**Главный сценарий:**
1. Открыл лендинг → заполнил форму → нажал «Оплатить и создать Паспорт жизни».
2. Перешёл на страницу оплаты BePaid → оплатил → вернулся на сайт.
3. Видит экран прогресса с этапами и прогресс-баром (не пустой экран!).
4. В фоне: бэкенд РБ убедился, что оплата прошла → отправил данные в польский API → получил JSON → собрал PDF.
5. Видит результат на сайте → скачивает PDF. При повторном заходе может скачать снова.
6. Если ошибка — видит понятную страницу FAIL.

**Экраны (во фронте уже есть основа):**
- Лендинг + форма — сбор данных, валидация, кнопка оплаты.
- Экран прогресса — этапы, прогресс-бар, проценты, анимация.
- Экран результата (SUCCESS) — превью паспорта + кнопка «Скачать PDF».
- Экран ошибки (FAIL) — понятное сообщение + что делать.

## 4. Что внутри (данные и сервисы)

**Хранит (сущность «Заказ»):**
- данные формы: имя, дата рождения, время рождения (или флаг «не знаю»), место рождения, 5 событий;
- идентификатор заказа;
- статус оплаты + дата/время;
- что отправили в польский API и какой JSON получили + дата/время;
- статус результата (SUCCESS / FAIL), дата генерации PDF, путь к PDF, текст ошибки (если была);
- журнал всех шагов (логи).

**Где живут данные:** советую **SQLite** — это «база данных в одном файле», не требует отдельного сервера, ставится сама. Для этого проекта её хватает с запасом и она не ломается. (В ТЗ это «Вариант 1 — база данных»; если Виктория выберет JSON-файлы — тоже допустимо, но SQLite надёжнее и проще в работе.)

**Внешние сервисы:**
- **BePaid** — оплата. Нужны ключи магазина (shop id + secret key) из личного кабинета BePaid. Подтверждение оплаты — **только** по вебхуку с проверкой подписи, не по факту возврата пользователя.
- **ИИ-модель** — какую именно, решает заказчик; код пишем так, чтобы модель **менялась через настройку**, без переделки логики.
- Между бэкендом РБ и польским API — **общий секрет/подпись** запроса + защита от повторной отправки (по идентификатору заказа).

**Договорённость о JSON между РБ и Польшей** (заложим простую и расширяемую структуру; точное «нутро» паспорта определится, когда будет шаблон промпта Виктории):
```json
{
  "status": "SUCCESS",
  "order_id": "abc-123",
  "passport": {
    "title": "Паспорт жизни",
    "sections": [
      { "heading": "Заголовок раздела", "body": "Текст раздела..." }
    ]
  },
  "error": null
}
```

## 5. На чём делаем (стек и инструмент)

**Стек (всё на одном языке — JavaScript, чтобы не распыляться):**
- **Лицо (фронт):** оставляем твой React + Tailwind как есть, только подключаем к бэкенду.
- **Кухня (бэкенд РБ):** Node.js + Express — простой сервер, тот же язык, что и фронт.
- **ИИ-цех (польский API):** тоже Node.js + Express — отдельный сервис.
- **Данные:** SQLite (файл-база, без отдельного сервера).
- **PDF:** генерируем из HTML-шаблона паспорта (библиотека рендера HTML→PDF). Так PDF выглядит так же, как визуальный паспорт на сайте.
- **Секреты:** в `.env`-файлах, никогда не в репозитории.

**Инструмент: Google Antigravity** — это агентный редактор кода (как Cursor/Windsurf, на базе Gemini 3). Ты ставишь задачу — агент сам планирует, пишет по нескольким файлам, запускает, проверяет в браузере. Подходит под этот проект, потому что тут реальный бэкенд и несколько сервисов, а не просто экранчик.

Практика под Antigravity:
- Открываешь **папку-воркспейс** (workspace) и работаешь внутри неё.
- Для серьёзных шагов включай **Planning mode** (агент сначала составит план/артефакты — удобно проверить логику до того, как он напишет код). Для мелочей — Fast mode.
- Для **шага с оплатой** используй режим с ручным подтверждением (review): платежи — место, где ошибаться нельзя, лучше одобрять действия агента вручную.

**Предупреждения (грабли):**
- Antigravity сейчас в публичном превью и **обрабатывает код на серверах Google.** Сам код — не страшно, но **не вставляй в промпты реальные секретные ключи и реальные данные клиентов.** Ключи — только в `.env` на сервере.
- BePaid сначала тестируй в **тестовом режиме** (test-флаг), реальные деньги — в самом конце.
- Доступ к ИИ-моделям и оплате может требовать карты/региона — проверь заранее, что ключи у тебя есть и работают.

## 6. План по этапам

Каждый этап заканчивается **проверяемым результатом**. Не переходи к следующему, пока текущий не работает. Сохраняй рабочую версию (git-коммит) после каждого удачного этапа.

**Этап 1 — Каркас и подключение фронта.** Заводим структуру проекта (папка фронта + папка бэкенда РБ), бэкенд отдаёт существующий лендинг, есть «живой» проверочный адрес. → *В конце работает:* проект открывается локально, лендинг показывается, бэкенд отвечает «я жив».

**Этап 2 — Сквозной путь на заглушке (без оплаты и ИИ).** Отправка формы → бэкенд создаёт заказ → сразу возвращает фейковый JSON-паспорт → бэкенд делает из него настоящий PDF → фронт показывает прогресс → результат → скачивание. → *В конце работает:* весь путь «форма → PDF → скачать» проходит на фейковых данных.

**Этап 3 — Визуальный паспорт и нормальный PDF.** Доводим HTML-шаблон паспорта и PDF до приличного вида (всё ещё на фейковом JSON). → *В конце работает:* PDF выглядит как настоящий паспорт, не «черновик».

**Этап 4 — Реальная оплата BePaid.** Вместо мгновенной заглушки вставляем оплату: форма → создание платежа → переход на BePaid → подтверждение по вебхуку (с проверкой подписи) → только потом запуск обработки (пока с фейковым JSON). → *В конце работает:* в тестовом режиме без оплаты дальше не пускает, после оплаты путь идёт дальше.

**Этап 5 — Польский API как отдельный сервис (заглушка ИИ).** Поднимаем отдельный сервис: принимает защищённый запрос, отдаёт структурированный JSON — пока с захардкоженным «ответом ИИ». Бэкенд РБ начинает звать его вместо своей внутренней заглушки. → *В конце работает:* два сервиса общаются, РБ получает JSON из настоящего польского сервиса.

**Этап 6 — Реальная ИИ-модель.** В польском API меняем заглушку на настоящий вызов ИИ + шаблон промпта (от Виктории), который лежит в отдельном редактируемом файле. → *В конце работает:* реальный ИИ генерирует содержимое паспорта.

**Этап 7 — Безопасность, логирование, отказоустойчивость, финал.** Подпись/секрет между сервисами, защита от повторной отправки, полное логирование на обеих сторонах, обработка ошибок + возможность повторной обработки, страницы SUCCESS/FAIL. → *В конце работает:* ошибки не теряются и логируются, запрос можно переобработать, доступ защищён.

## 9. Советы по работе с Antigravity

- **Бриф — в Rules, не в каждый промпт.** Сохрани промпт-бриф как правило проекта (Customizations → Rules). Тогда агент держит контекст постоянно, и каждый шаговый промпт можно делать короче.
- **Маленькими шагами + git после каждого удачного этапа.** Закоммитил рабочую версию → пошёл дальше. Если агент что-то сломал на следующем шаге — откатишься без потерь.
- **Planning mode для крупных шагов, ручное подтверждение для оплаты.** На этапах 4 (оплата) держи строгий контроль — одобряй действия агента вручную, это критичная зона.
- **Проверяй после каждого промпта по строке «Результат».** Не идёшь дальше, пока текущий чекпоинт реально не работает в браузере.
- **Не вставляй в промпты реальные ключи и данные клиентов.** Код Antigravity обрабатывает на серверах Google — секреты держи только в `.env` на своей стороне.
- **Если агент «застрял» или начал ломать рабочее** — переформулируй промпт конкретнее и разбей на более мелкие шаги. Один промпт = одно понятное изменение.
